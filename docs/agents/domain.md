# 도메인 문서

엔지니어링 스킬이 코드베이스를 탐색할 때 이 저장소의 도메인 문서를 어떻게 읽어야 하는지에 대한 규칙.

## 탐색 전에 읽을 것

- 저장소 루트의 **`CONTEXT.md`**, 또는
- 루트에 **`CONTEXT-MAP.md`**가 있으면 그 파일이 가리키는 컨텍스트별 `CONTEXT.md`를 주제와 관련된 것만 읽는다.
- **`docs/adr/`** — 작업할 영역과 관련된 ADR을 읽는다. 멀티 컨텍스트 저장소에서는 `src/<context>/docs/adr/`의 컨텍스트별 결정도 확인한다.

위 파일들이 없으면 **조용히 진행한다**. 부재를 보고하거나 미리 만들자고 제안하지 않는다. 생산자 스킬(`/grill-with-docs`)이 용어나 결정이 실제로 정해질 때 게으르게(lazy) 만든다.

## 파일 구조

단일 컨텍스트 저장소 (대부분의 경우):

```
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-event-sourced-orders.md
│   └── 0002-postgres-for-write-model.md
└── src/
```

멀티 컨텍스트 저장소 (루트에 `CONTEXT-MAP.md` 존재):

```
/
├── CONTEXT-MAP.md
├── docs/adr/                          ← 시스템 전역 결정
└── src/
    ├── ordering/
    │   ├── CONTEXT.md
    │   └── docs/adr/                  ← 컨텍스트별 결정
    └── billing/
        ├── CONTEXT.md
        └── docs/adr/
```

## 용어집의 어휘를 사용한다

산출물에서 도메인 개념을 언급할 때(이슈 제목, 리팩터 제안, 가설, 테스트 이름 등)는 `CONTEXT.md`에 정의된 용어를 그대로 쓴다. 용어집이 명시적으로 피하는 동의어로 흘러가지 말 것.

필요한 개념이 아직 용어집에 없다면, 그건 신호다 — 프로젝트가 쓰지 않는 언어를 만들어내고 있거나(재고할 것), 진짜로 빠진 부분이거나(`/grill-with-docs`에 메모할 것).
